# 🔧 Архитектурные проблемы и решения

---

## 📋 Выявленные проблемы

### Проблема 1: God Object паттерн в GameState

**Описание:**  
`GameState` содержит слишком много методов, реализованных в разных файлах (state.rs, logic.rs, scoring.rs, render.rs). Это делает сложным понимание, где находится какой функционал.

**Файлы:** `src/game/state.rs`, `src/game/mod.rs` (97-250)

**Влияние на архитектуру:**
- 🔴 Высокая связанность (coupling)
- 🔴 Сложность тестирования
- 🟡 Затруднение навигации по коду
- 🟡 Нарушение SRP (Single Responsibility Principle)

---

### Варианты решения Проблемы 1

#### Вариант 1A: Разделение GameState на несколько структур (RECOMMENDED)

**Описание:**  
Разделить монолитный `GameState` на несколько структур, каждая отвечающая за свою область:
- `GamePlayState` — физика, текущие фигуры, поле
- `GameScore` — очки, уровень, линии
- `GameStats` — статистика (уже существует)

**Преимущества:**
- ✅ Четкое разделение ответственности
- ✅ Легче тестировать части отдельно
- ✅ Понятнее интерфейсы функций
- ✅ Легче расширять функциональность

**Недостатки:**
- ❌ Требует рефакторинга всего кода
- ❌ Может нарушить обратную совместимость
- ❌ Больше боilerplate кода

**Трудоемкость:** 🔴 ВЫСОКАЯ (4-6 часов)

**Код:**
```rust
pub struct GamePlayState {
    pub curr_shape: Tetromino,
    pub next_shape: Tetromino,
    pub hold_shape: Option<Tetromino>,
    pub blocks: Grid,
    pub fall_spd: f32,
    pub land_time_ms: f64,
}

pub struct GameScore {
    pub score: u128,
    pub level: u32,
    pub lines: u32,
    pub soft_drop_distance: u128,
}

pub struct GameState {
    pub play: GamePlayState,
    pub score: GameScore,
    pub stats: GameStats,
    pub mode: GameMode,
    pub curr_animation_frame: u32,
}
```

#### Вариант 1B: Использование трейтов для разделения ответственности

**Описание:**  
Создать трейты для разных аспектов GameState:
- `Updateable` — для логики обновления
- `Drawable` — для отрисовки
- `Scorable` — для подсчета очков

**Преимущества:**
- ✅ Не требует изменения структуры
- ✅ Четкие интерфейсы между компонентами
- ✅ Легче добавить новые компоненты
- ✅ Поддержка полиморфизма

**Недостатки:**
- ❌ Требует больше boilerplate кода
- ❌ Может быть излишней абстракцией (YAGNI)
- ❌ Сложнее для начинающих

**Трудоемкость:** 🟡 СРЕДНЯЯ (3-4 часа)

**Код:**
```rust
pub trait Updateable {

}
```

### Проблема 2: Тесная связь между game/ подмодулями

**Описание:**  
Каждый подмодуль (logic.rs, scoring.rs, render.rs) тесно зависит от GameState. Используется паттерн `&mut GameState` везде, что затрудняет тестирование в изоляции.

**Файлы:** `src/game/logic.rs`, `src/game/scoring.rs`, `src/game/render.rs`

**Влияние на архитектуру:**
- 🔴 Высокий coupling между модулями
- 🟡 Сложно переиспользовать компоненты
- 🟡 Трудно расширять без изменения GameState
- 🟡 Затрудняется unit-тестирование

### Варианты решения Проблемы 2

**Описание:**  
Вместо передачи `&mut GameState`, передавать только нужные данные через специальные объекты.

**Преимущества:**
- ✅ Минимальная связанность
- ✅ Четкие интерфейсы
- ✅ Легче тестировать
- ✅ Явное указание зависимостей

**Недостатки:**
- ❌ Больше boilerplate кода
- ❌ Необходимо синхронизировать DTO с GameState

**Трудоемкость:** 🟡 СРЕДНЯЯ (3-4 часа)

**Код:**
```rust

pub struct PhysicsOutput {
```

### Проблема 3: Недостаток абстракций для IO

**Описание:**  
`Canvas` и `KeyReader` напрямую используются из всех модулей. Нет трейтов для абстрагирования деталей реализации терминала. Сложно писать unit-тесты без настоящего терминала.

**Файлы:** `src/io.rs`, `src/game/logic.rs`, `src/game/render.rs`, `src/main.rs`

**Влияние на архитектуру:**
- 🟡 Сложность unit-тестирования игровой логики
- 🟡 Привязка к конкретной реализации терминала
- 🟡 Сложность создания mock-объектов для тестов

---

### Варианты решения Проблемы 3

#### Вариант 3A: Создать трейты для IO (RECOMMENDED)

**Описание:**  
Создать общие трейты `InputReader` и `OutputWriter` для абстрагирования деталей IO.

**Преимущества:**
- ✅ Легко создавать mock-объекты для тестов
- ✅ Логика не зависит от конкретной реализации
- ✅ Легче портировать на другие платформы
- ✅ Лучше для unit-тестирования

**Недостатки:**
- ❌ Требует рефакторинга
- ❌ Dynamic dispatch может быть медленнее
- ❌ Больше boilerplate кода

**Трудоемкость:** 🟡 СРЕДНЯЯ (2-3 часа)

**Код:**
```rust
pub trait InputReader {
    fn read_key(&mut self) -> Option<u8>;
}

pub trait OutputWriter {
    fn draw_string(&mut self, s: &str, pos: (u16, u16), color: &dyn Color);
    fn flush(&mut self);
}

pub struct MockInputReader {
    keys: Vec<u8>,
}

impl InputReader for MockInputReader {
    fn read_key(&mut self) -> Option<u8> {
        self.keys.pop()
    }
}
```

### Проблема 5: Обработка ошибок через assert!

**Описание:**  
Используются `unwrap()`, `assert!` без контекста. Нет систематической обработки ошибок через `Result`. Паникует вместо graceful shutdown.

**Файлы:** везде в проекте

**Влияние на архитектуры:**
- 🔴 Непредсказуемые crashes
- 🟡 Сложность отладки
- 🟡 Нарушение принципа Fail-Safe
- 🟡 Плохой UX при ошибках

---

### Варианты решения Проблемы 5

#### Вариант 5A: Внедрить Result-based обработку (RECOMMENDED)

**Описание:**  
Создать свой тип `GameError` и использовать `Result<T, GameError>` для обработки ошибок.

**Преимущества:**
- ✅ Graceful error handling
- ✅ Лучше для UX
- ✅ Понятнее где ошибки
- ✅ Легче отлаживать

**Недостатки:**
- ❌ Требует рефакторинга сигнатур функций
- ❌ Больше boilerplate кода
- ❌ Может снизить производительность (?)

**Трудоемкость:** 🟡 СРЕДНЯЯ (3-4 часа)

**Код:**
```rust
pub enum GameError {
    ConfigLoadError(String),
    CanvasError(String),
    IOError(std::io::Error),
    SerializationError(String),
}

pub type GameResult<T> = Result<T, GameError>;

fn load_config() -> GameResult<ControlsConfig> {
    confy::load("tetris-cli", "config")
        .map_err(|e| GameError::ConfigLoadError(e.to_string()))
}
```
