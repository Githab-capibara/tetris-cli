## 📊 Обзор архитектуры

Tetris CLI использует **модульную архитектуру** с чётким разделением ответственности между компонентами. Проект следует принципам **SOLID** и **DRY**, обеспечивая поддерживаемость и расширяемость кода.

### Ключевые принципы

- **Модульность**: каждый модуль отвечает за одну область функциональности
- **Инкапсуляция**: скрытие внутренней реализации, публичный API через mod.rs
- **Безопасность**: защита данных через хеширование и валидацию
- **Производительность**: 60 FPS стабильно через оптимизированный игровой цикл
- **Тестируемость**: 1823 модульных и интеграционных тестов
- **Отсутствие циклических зависимостей**: общие типы вынесены в отдельный модуль types.rs
- **Разделение ответственности**: application layer, игровая логика, отрисовка и состояние разделены
- **Расширяемость**: GameModeTrait для добавления новых режимов без изменения ядра

## 📦 Модули проекта

### main.rs

**Назначение**: Точка входа в приложение и управление состоянием меню.

**Основные компоненты**:
- `MENU` — массив строк для отображения главного меню
- `LEADERBOARD_MENU` — меню таблицы лидеров
- `main()` — инициализация терминала, загрузка рекорда, цикл меню
- `get_player_name()` — запрос имени для таблицы лидеров
- `show_leaderboard()` — отображение топ-5 рекордов
- `show_game_stats()` — показ статистики после игры

**Поток выполнения**:
1. Загрузка рекорда из конфигурации
2. Загрузка и валидация таблицы лидеров
3. Проверка размера терминала
4. Инициализация Canvas и KeyReader
5. Цикл отрисовки меню (60 FPS)
6. Обработка нажатий:
   - `Enter` — запуск классической игры
   - `R` — запуск режима спринт
   - `M` — запуск режима марафон
   - `L` — отображение таблицы лидеров
   - `Backspace` — выход
7. После игры: отображение статистики, сохранение рекорда

#### state.rs

**Назначение**: Структуры состояния игры и константы.

**Основные компоненты**:
- `GameState` — основное состояние игры (поля публичные для доступа из подмодулей)
- `GameStats` — статистика игры (время, фигуры, комбо)
- `GameMode` — режим игры (Classic, Sprint, Marathon)
- `Achievement` — система достижений (заготовка)
- Константы: `FPS`, `INITIAL_FALL_SPD`, `LINE_SCORES`, `COMBO_BONUS`, и др.

**Пример использования**:
```rust
use tetris_cli::game::GameState;

let mut game = GameState::new();
game.play(&mut cnv, &mut inp, &high_score_display);
```

#### cycle.rs

**Назначение**: Разделение игрового цикла на отдельные фазы.

**Основные компоненты**:
- `run_game_loop()` — основной игровой цикл
- `handle_input_phase()` — фаза обработки ввода
- `update_phase()` — фаза обновления состояния
- `render_phase()` — фаза отрисовки
- `handle_game_over()` — обработка конца игры

**Разделение фаз**:
```rust
// Основной цикл разделён на независимые фазы:
loop {
    handle_fps_control();  // Поддержание 60 FPS
    process_input();       // Обработка ввода
    update_state();        // Обновление логики
    render();              // Отрисовка кадра
    handle_animations();   // Анимации
}
```

**Преимущества**:
- Улучшена тестируемость каждой фазы отдельно
- Упрощена поддержка и расширение
- Соблюдение принципа единственной ответственности

#### cache.rs

**Назначение**: Кэширование строк отрисовки для снижения аллокаций.

**Основные компоненты**:
- `RenderCache` — структура для кэширования строк отрисовки
  - `cached_score_str` — закэшированная строка счёта
  - `cached_level_str` — закэшированная строка уровня
  - `cached_lines_str` — закэшированная строка линий
  - `cached_high_score_str` — закэшированная строка рекорда
  - `cached_combo_str` — закэшированная строка комбо
  - `cached_timer_str` — закэшированная строка таймера
  - `last_cached_*` — последние закэшированные значения

**Кэшируемые данные**:
- Счёт (`cached_score_str`)
- Уровень (`cached_level_str`)
- Линии (`cached_lines_str`)
- Рекорд (`cached_high_score_str`)
- Комбо (`cached_combo_str`)
- Таймер (`cached_timer_str`)

**Преимущества**:
- Уменьшено количество полей в `GameState`
- Централизованное управление кэшированием
- Упрощение тестирования кэширования
- Обновление только при изменении данных
- Снижение количества аллокаций при отрисовке
- Улучшение производительности на ~10-15%

### tetromino.rs

**Назначение**: Определение типов фигур и их поведения.

**Типы фигур (ShapeType)**:

| Тип | Название | Цвет | Координаты |
|-----|----------|------|------------|
| T | T-образная | Пурпурный (Magenta) | (-1,0), (0,0), (1,0), (0,1) |
| L | L-образная | Жёлтый (Yellow) | (-1,-1), (0,-1), (0,0), (0,1) |
| J | J-образная | Синий (Blue) | (1,-1), (0,-1), (0,0), (0,1) |
| S | S-образная | Зелёный (Green) | (0,-1), (0,0), (1,0), (1,1) |
| Z | Z-образная | Светло-красный (LightRed) | (0,-1), (0,0), (-1,0), (-1,1) |
| O | Квадрат | Светло-жёлтый (LightYellow) | (0,0), (1,0), (0,1), (1,1) |
| I | Линия | Голубой (Cyan) | (0,-1), (0,0), (0,1), (0,2) |

**Bag Generator**:
- Система 7-bag гарантирует равномерное распределение фигур
- Каждые 7 фигур содержат все 7 типов
- Использует алгоритм Fisher-Yates для перемешивания
- **Оптимизация**: используется `reserve()` + `extend_from_slice()` для уменьшения аллокаций

pub struct PathValidator;

pub enum PathError {
    TooLong { max: usize, actual: usize },
    InvalidCharacter { char: char, position: usize },
    SymlinkDetected,
    PathTraversalAttempt,
}

pub enum PathErrorKind {
    Length,
    Character,
    Symlink,
    Traversal,
}

impl PathValidator {
    pub fn validate(path: &str) -> Result<(), PathError>;
    pub fn validate_length(path: &str, max: usize) -> Result<(), PathError>;
    pub fn validate_characters(path: &str) -> Result<(), PathError>;
    pub fn validate_no_symlink(path: &str) -> Result<(), PathError>;
    pub fn validate_no_traversal(path: &str) -> Result<(), PathError>;
}
```

**Преимущества**:
- Централизованная валидация путей
- Типизированные ошибки для каждой категории проблем
- Устранение дублирования кода
- Упрощение тестирования валидации

### Июнь 2026: Версия 23.96.27 — Архитектурный рефакторинг

В версии 23.96.27 реализован комплексный архитектурный рефакторинг с выделением подмодулей и улучшением модульной структуры.

#### Реализованные улучшения

##### 1. Выделение game/logic/ — подмодули игровой логики

**Проблема**: Модуль logic.rs содержал смешанную ответственность (обработка ввода, физика, коллизии, вращение) в одном файле.

**Решение**: Создан модуль `game/logic/` с разделением ответственности:

```
src/game/logic/
├── mod.rs      # Публичный API, re-export функций
├── input.rs    # Обработка ввода пользователя
├── physics.rs  # Физика и гравитация
├── collision.rs# Проверка коллизий
├── rotation.rs # Вращение с wall kick
└── update.rs   # Обновление состояния игры
```

**Преимущества**:
- Разделение ответственности по принципу Single Responsibility
- Улучшенная тестируемость каждого компонента отдельно
- Упрощённая навигация по коду
- Возможность независимого тестирования физики, коллизий и вращения

##### 2. Выделение game/scoring/ — подмодули системы очков

**Проблема**: Модуль scoring.rs смешивал логику поиска линий, начисления очков и комбо-системы.

**Решение**: Создан модуль `game/scoring/` с разделением:

```
src/game/scoring/
├── mod.rs      # Публичный API, re-export функций
├── lines.rs    # Поиск и удаление линий
├── points.rs   # Начисление очков
└── combo.rs    # Комбо-логика
```

**Преимущества**:
- Чёткое разделение логики линий, очков и комбо
- Упрощённое добавление новых механик начисления очков
- Улучшенная тестируемость каждой системы отдельно
- Снижение связанности между компонентами

##### 3. Создание validation/ — модуль валидации

**Проблема**: Функции валидации были разбросаны по highscore.rs и controls.rs.

**Решение**: Создан отдельный модуль `validation/`:

```
src/validation/
├── mod.rs      # Публичный API валидации
├── name.rs     # Валидация имен игроков
└── path.rs     # Валидация путей к конфигурации
```

**Основные компоненты**:

**name.rs**:
- `validate_player_name()` — комплексная валидация имени
- `sanitize_player_name()` — очистка и нормализация
- Whitelist-фильтрация разрешённых символов
- Защита от Unicode-атак
- Оптимизация производительности (~15-20%)

**path.rs**:
- `validate_config_path()` — проверка пути конфигурации
- `validate_no_traversal()` — защита от Path Traversal
- `validate_no_symlink()` — проверка на симлинки
- `validate_length()` — проверка длины пути
- `validate_characters()` — проверка допустимых символов

**Преимущества**:
- Централизованная валидация данных
- Повторное использование функций валидации
- Улучшенная безопасность через типизированные ошибки
- Снижение дублирования кода

##### 4. Улучшение инкапсуляции GameState

**Проблема**: Поля `GameState` оставались публичными, что нарушало инкапсуляцию.

**Решение**:

- Подготовлена основа для будущей приватности
- Улучшенная читаемость структуры

##### 5. GameModeTrait вместо enum GameMode

**Преимущества**:
- Снижение сложности кода

- Сохранение необходимого уровня безопасности

- Снижение связанности между модулями

### Принципы, которым следует проект

#### SOLID
- **Single Responsibility** — каждый модуль отвечает за одну область (game/, io.rs, tetromino.rs)
- **Open/Closed** — расширение через новые модули без изменения существующих
- **Liskov Substitution** — типы взаимозаменяемы (Direction enum)
- **Interface Segregation** — узкие интерфейсы (Canvas, KeyReader)
- **Dependency Inversion** — зависимость от абстракций (types.rs как leaf module)

#### DRY/KISS/YAGNI
- **DRY** — общие функции вынесены в crypto.rs, types.rs
- **KISS** — простая архитектура без избыточных абстракций
- **YAGNI** — удаление неиспользуемого кода (deprecated функции)

    fn check_win_condition(&self, lines: u32) -> bool;
    fn get_target_lines(&self) -> Option<u32>;
    fn name(&self) -> &str;
}

// game/modes.rs
pub struct ClassicMode;
pub struct SprintMode { target: u32 }
pub struct MarathonMode { target: u32 }

impl GameModeTrait for ClassicMode { ... }
impl GameModeTrait for SprintMode { ... }
impl GameModeTrait for MarathonMode { ... }
```

**Преимущества**:
- Можно добавить новый режим без изменения существующего кода
- Режимы реализуются как отдельные структуры
- Поддержка динамической диспетчеризации через `Box<dyn GameModeTrait>`
- Соблюдение Open/Closed Principle

**Пример добавления нового режима**:
```rust
struct UltraMode { target: u32 }
impl GameModeTrait for UltraMode {

    }
    fn get_target_lines(&self) -> Option<u32> {
        Some(self.target)
    }
    fn name(&self) -> &str { "Ultra" }
}
```

### Типобезопасные обёртки (Primitive Obsession Fix)

**Проблема:** Использование примитивных типов u128, u32 для очков, уровня, линий приводило к логическим ошибкам и отсутствию инвариантов.

**Решение:** Создан модуль `game/types.rs` с типами-обёртками:

```rust
pub struct Score(u128);      // Очки игрока
pub struct Level(u32);       // Уровень (минимум 1)
pub struct LinesCount(u32);  // Количество линий
```

**Возможности:**
- Saturating операции для защиты от переполнения
- Инварианты (Level >= 1)
- Методы для типичных операций (add, multiply, reset)
- Реализация Display, Default, From/Into

**Преимущества:**
- Типобезопасность (невозможно перепутать Score и LinesCount)
- Инкапсуляция логики (сатурирующее сложение)
- Самодокументируемый код
