## 📊 Обзор архитектуры

Tetris CLI использует **модульную архитектуру** с чётким разделением ответственности между компонентами. Проект следует принципам **SOLID** и **DRY**, обеспечивая поддерживаемость и расширяемость кода.

### Ключевые принципы

- **Модульность**: каждый модуль отвечает за одну область функциональности
- **Инкапсуляция**: скрытие внутренней реализации, публичный API через mod.rs
- **Безопасность**: защита данных через хеширование и валидацию
- **Производительность**: 60 FPS стабильно через оптимизированный игровой цикл, массив blocks на стеке
- **Тестируемость**: 1580+ модульных и интеграционных тестов + бенчмарки производительности, включая тесты для 32 исправлений аудита кода
- **Отсутствие циклических зависимостей**: общие типы вынесены в отдельный модуль types.rs

### types.rs

**Назначение**: Общие типы и перечисления для использования во всех модулях.

Этот модуль был создан для устранения циклических зависимостей между модулями game.rs и tetromino.rs.

**Основные компоненты**:
- `Direction` — направление движения фигуры (Left, Right, Down)
- `RotationDirection` — направление вращения фигуры (Clockwise, CounterClockwise)
- `UpdateEndState` — состояние завершения обновления (Quit, Lost, Continue, Pause, Won)

**Пример использования**:
```rust
use tetris_cli::types::Direction;

let dir = Direction::Left;
match dir {
    Direction::Left => println!("Движение влево"),
    Direction::Right => println!("Движение вправо"),
    Direction::Down => println!("Движение вниз"),
}
```

**Зависимости**:
- Не зависит от других модулей проекта
- Используется в: game.rs, tetromino.rs, main.rs

### crypto.rs

**Назначение**: Криптографические утилиты для всего проекта.

Этот модуль был создан для устранения дублирования кода хеширования между модулями highscore.rs и controls.rs.

**Основные компоненты**:
- `hash()` — хеширование BLAKE3 (256 бит)
- `generate_salt()` — генерация случайной соли (256 бит)
- `hmac()` — вычисление HMAC-подобной подписи
- `verify_hmac()` — проверка HMAC подписи

**Пример использования**:
```rust
use tetris_cli::crypto::{hash, generate_salt, hmac};

// Хэширование
let h = hash("данные");

// Генерация соли
let salt = generate_salt();

// HMAC подпись
let signature = hmac("ключ", "данные");
```

**Зависимости**:
- Зависит от: blake3, hex, rand
- Используется в: highscore.rs, controls.rs

### tetromino.rs

**Назначение**: Определение типов фигур и их поведения.

**Типы фигур (ShapeType)**:

| Тип | Название | Цвет | Координаты |
|-----|----------|------|------------|
| T | T-образная | Пурпурный (Magenta) | (-1,0), (0,0), (1,0), (0,1) |
| L | L-образная | Жёлтый (Yellow) | (-1,-1), (0,-1), (0,0), (0,1) |
| J | J-образная | Синий (Blue) | (1,-1), (0,-1), (0,0), (0,1) |
| S | S-образная | Зелёный (Green) | (0,-1), (0,0), (1,0), (1,1) |
| Z | Z-образная | Светло-красный (LightRed) | (0,-1), (0,0), (-1,0), (-1,1) |
| O | Квадрат | Светло-жёлтый (LightYellow) | (0,0), (1,0), (0,1), (1,1) |
| I | Линия | Голубой (Cyan) | (0,-1), (0,0), (0,1), (0,2) |

**Bag Generator**:
- Система 7-bag гарантирует равномерное распределение фигур
- Каждые 7 фигур содержат все 7 типов
- Использует алгоритм Fisher-Yates для перемешивания
- **Оптимизация**: используется `reserve()` + `extend_from_slice()` в методе `fill_bag()` для уменьшения аллокаций
- **API**: используется `blake3::hash()` для хеширования вместо ручного кодирования

**Методы Tetromino**:
- `select()` — случайный выбор новой фигуры
- `from_bag()` — создание фигуры из Bag Generator
- `rotate()` — вращение фигуры на 90°
  - Удалена избыточная проверка `Dir::Down` для упрощения логики
- `get_bag()` — получение мешка с фигурами
  - Атрибут `#[must_use]`
- `get_index()` — получение индекса фигуры
  - Атрибут `#[must_use]`

### highscore.rs

**Назначение**: Сохранение и загрузка рекордов с защитой от подделки.

**Структура SaveData**:
```rust
pub struct SaveData {
    pub high_score: u64,           // Значение рекорда
    pub high_score_salt: String,   // Случайная соль для хеширования
    pub high_score_hash: String,   // Хеш рекорда с солью

```

**Структура LeaderboardEntry**:
```rust
pub struct LeaderboardEntry {
    pub name: String,    // Имя игрока (публичное для чтения)
    pub score: u64,      // Значение рекорда (публичное для чтения)
    salt: String,        // Соль (приватное)

```

**Структура Leaderboard**:
```rust
pub struct Leaderboard {
    pub entries: Vec<LeaderboardEntry>,  // Вектор записей (максимум 5)
    pub last_write_time: Option<Instant>, // Время последней записи
    pub write_cooldown: Duration,         // Cooldown между записями (5 сек)

```

**Методы**:
- `load_config()` — загрузка из файла конфигурации (использует `verify_and_get_score().unwrap_or(0)`)
- `from_value()` — создание новой записи с хешем
- `save_value()` — сохранение рекорда
- `verify_and_get_score() -> Option<u64>` — проверка целостности рекорда (возвращает `Some(score)` или `None`)
  - Атрибут `#[must_use]` — результат должен быть использован
- `assert_hs()` — проверка целостности рекорда (устарел, используйте `verify_and_get_score()`)
- `get_random_hash()` — генерация случайной соли
- `get_hash()` — вычисление хеша строки (BLAKE3)
- `add_score()` — добавление рекорда в таблицу
- `validate()` — валидация записей
  - Атрибут `#[must_use]` — результат должен быть использован
- `save()` — сохранение таблицы лидеров
- `score()` — получение значения рекорда
  - Атрибут `#[must_use]`
- `is_valid()` — проверка валидности записи
  - Атрибут `#[must_use]`
- `get_best_score()` — получение лучшего рекорда
  - Атрибут `#[must_use]`
- `len()` — получение количества записей
  - Атрибут `#[must_use]`
- `is_empty()` — проверка пустоты таблицы
  - Атрибут `#[must_use]`

**Защита от подделки**:
Рекорд сохраняется вместе с криптографической солью и хешем. При загрузке хеш пересчитывается и сравнивается с сохранённым. Если хеши не совпадают, рекорд сбрасывается в 0.

**Логирование попыток подделки**:
При обнаружении несоответствия хеша в лог выводится предупреждение о попытке подделки рекорда.

**Rate Limiting**:
- Cooldown 6 секунд между записями в таблицу лидеров (10 записей в минуту)
- Защита от DoS через спам рекордов
- Отслеживание времени последней записи (`last_write_time`)
- Отключается в тестах через `#[cfg(not(test))]`

**Whitelist валидация имён**:
- Разрешены только безопасные символы: буквы, цифры, '_', '-', ' ', '.'
- Пустые имена после фильтрации заменяются на "Anonymous"

**Оптимизация**:
- Используется `String::with_capacity()` + `write!()` вместо `format!()` для улучшения производительности
- Уменьшение аллокаций памяти при форматировании строк

- `const fn` — константные функции для инициализации конфигурации

**Безопасность**:
- Усилена защита от Path Traversal с использованием `canonicalize()`
- Валидация путей (запрет абсолютных путей и `..`)
- Проверка на дубликаты клавиш
- Фильтрация недопустимых значений

## 🎮 Игровой цикл (детали)

### Частота обновления
- **FPS**: 60 кадров в секунду
- **Интервал кадра**: ~16.67 мс (`FRAME_DELAY_MS`)

### Физика падения
- Начальная скорость: 0.9 блоков/секунду
- Прирост скорости: +0.05 за каждую заполненную линию
- Плавное движение с интерполяцией по времени

### Таймер приземления
- Задержка после касания: 0.1 секунды
- Даёт время на перемещение фигуры перед фиксацией

### Wall Kick
- Система проверки столкновений при вращении
- Константа `WALL_KICK_OFFSETS` содержит смещения для проверки
- 5 позиций для проверки возможности вращения

## 🔒 Безопасность

### Защита рекордов

1. **BLAKE3 хеширование** — криптографически стойкая функция
2. **Соль для хешей** — 32 байта случайных данных из OS RNG
3. **Валидация имён** — фильтрация опасных символов
4. **Path traversal защита** — в `controls.rs`
5. **Rate limiting** — cooldown 6 секунд между записями (10 записей в минуту)
6. **Integrity check** — проверка хеша при загрузке
7. **Массив на стеке** — `[[i8; GRID_WIDTH]; GRID_HEIGHT]` вместо `Box<...>`

### Инкапсуляция

- Приватные поля в `LeaderboardEntry` (salt)
- Публичные геттеры для контролируемого доступа
- Валидация всех входных данных

### Проверки границ

- `debug_assert!` для безопасности cast в `usize`
- `.max(0.0)` для предотвращения отрицательных значений
- Проверки границ массивов перед доступом
