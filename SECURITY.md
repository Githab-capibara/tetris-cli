# Политика безопасности

## Обзор

Этот документ описывает меры безопасности, реализованные в tetris-cli, и процесс сообщения об уязвимостях.

## Реализованные меры безопасности

### 1. Защита конфигурации управления

#### HMAC подпись
- Конфигурация управления подписывается keyed hash (HMAC) при сохранении
- При загрузке проверяется целостность подписи
- Защита от подделки конфигурации через редактирование JSON файла

#### Rate limiting
- Минимальный интервал между сохранениями: 60 секунд
- Предотвращает износ диска при частых вызовах
- Максимум 3 записи на одного игрока в таблице лидеров

#### Валидация путей
- Запрет абсолютных путей
- Защита от path traversal (`..`)
- Проверка через `O_NOFOLLOW` для защиты от symlink атак
- Валидация через централизованный `PathValidator`

### 2. Защита таблицы лидеров

#### Keyed hash с солью
- Каждая запись содержит уникальную соль
- Хэш вычисляется как `hash(salt + name + score)`
- Защита от подделки результатов

#### Валидация имени игрока
- Санитизация имени перед сохранением
- Запрет специальных символов и управляющих последовательностей
- Максимальная длина имени: 10 символов

#### TOCTOU защита
- Метод `score()` выполняет атомарную валидацию и возврат значения
- Валидация хэша для того же значения, которое возвращается
- **Ограничение:** структура НЕ потокобезопасна (см. документацию `LeaderboardEntry`)

### 3. Безопасность ввода/вывода

#### Drop guard для терминала
- Гарантированный сброс терминала при выходе или панике
- Показ курсора, возврат на главный экран
- Логирование критических ошибок через `eprintln!`

#### Обработка UTF-8
- Метод `get_key()` поддерживает только ASCII (0x00-0x7F)
- Метод `get_key_unicode()` поддерживает полный Unicode
- Невалидные UTF-8 последовательности отбрасываются
- Защита от переполнения буфера при чтении многобайтовых символов

### 4. Защита от переполнения

#### u128 для очков
- Все счётчики очков используют `u128`
- Предотвращение переполнения при больших значениях
- Максимальное значение: 2^128 - 1

#### Saturating операции
- Использование `saturating_add()` для прироста скорости
- Защита от переполнения при вычислениях

### 5. Инкапсуляция данных

#### Приватные поля
- Поля `GameState` имеют `pub(crate)` видимость
- Доступ через геттеры/сеттеры
- Контроль изменения состояния

#### Трейты доступа
- `GameBoardAccess` для контролируемого доступа к полю
- `ScoreAccess` для доступа к очкам (TODO)

## Сообщение об уязвимостях

### Как сообщить

Если вы обнаружили уязвимость безопасности, пожалуйста:

1. **НЕ создавайте публичный issue** на GitHub
2. Отправьте отчёт на email: [указать email]
3. Или создайте приватный security advisory на GitHub

### Что включать в отчёт

- Описание уязвимости
- Шаги для воспроизведения
- Версия tetris-cli
- Возможные последствия
- Рекомендации по исправлению (если есть)

### Время ответа

- Первоначальный ответ: в течение 48 часов
- План исправления: в течение 7 дней
- Патч: в течение 30 дней для критических уязвимостей

## Известные ограничения

### 1. Однопоточная безопасность

`LeaderboardEntry` НЕ является потокобезопасным:
- Методы валидации могут подвергаться TOCTOU в многопоточной среде
- Решение: используйте `Arc<Mutex<LeaderboardEntry>>` для многопоточного доступа

### 2. UTF-8 в get_key()

Метод `get_key()` не поддерживает многобайтовые символы:
- Возвращает `None` для кириллицы, emoji и других Unicode символов
- Решение: используйте `get_key_unicode()` для поддержки Unicode

### 3. Терминал-зависимость

Проект использует `termion` который работает только на Unix-системах:
- Не поддерживается Windows без WSL
- TODO: реализация `TerminalBackend` для кроссплатформенности
